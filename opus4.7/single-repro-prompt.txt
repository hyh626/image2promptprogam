I want to build an autoresearch loop following Andrej Karpathy's autoresearch
pattern (https://github.com/karpathy/autoresearch), but applied to a different
problem: given a reference image, discover a prompting strategy that causes
an image generator to reproduce it as faithfully as possible.

The methodology: a coding agent edits a single Python file
(`prompt_strategy.py`), runs a fixed benchmark, reads the score, and iterates.
Comparability across experiments depends on the harness being unchanging.

Do NOT generate a README.md — that is a separate artifact. Do NOT write the

## The stack

- VLM (writes the prompt from the image, called inside `prompt_strategy.py`):
  `gemini-3.1-flash-lite-preview`
- Image generator (called by the harness): `gemini-3.1-flash-image-preview`
  (Nano Banana 2). Fixed config: 1024×1024, defaults, no aspect ratio override.
- Semantic similarity: `gemini-embedding-2-preview` (multimodal, 3072-d, used
  for both original and regenerated images in the same shared space).

The driver agent that runs the loop is independent of the inner VLM. The
driver might be Claude Code / Codex / Gemini CLI; the inner VLM is always
Gemini Flash-Lite. Call this separation out explicitly in both docs.

## Why semantic embedding alone isn't enough — and the local metrics that fix it

Gemini embeddings capture semantic content but not visual fidelity. Two
photos that both depict "golden retriever on grass" can embed close together
while looking nothing alike. A driver agent will Goodhart any single metric.
So three additional locally-computed metrics are required, spanning roughly
orthogonal axes from the semantic one:

- **Structural:** DINOv2 ViT-B/14
  (`facebook/dinov2-base` via HuggingFace `transformers`, Apache 2.0)
  — self-supervised features capturing pose, layout, appearance.
  We use DINOv2 (Apache 2.0) rather than DINOv3 (Meta's custom
  commercial license) for licensing simplicity; for our pair-similarity
  use case the practical gap is small, since DINOv3's improvements are
  mostly on dense-prediction tasks.
- **Perceptual:** LPIPS with AlexNet backbone (`lpips` package) — texture
  and fine-detail match, trained against human similarity judgments.
- **Color:** HSV histogram with 8×8×8 bins, chi-square distance — explicit
  palette match, catches "right object, wrong colors" failures.

All four metrics are computed at canonical **448×448** (multiple of DINOv2
patch size 14, since 448/14 = 32) with PIL `LANCZOS` resize, so
framing/resolution differences don't pollute the signal.

CPU-only execution must work — no GPU required. Per 20-image experiment,
local metrics combined add ~10s on CPU; the bottleneck is the ~3–5 minutes
of Nano Banana 2 generation calls. Original-image features are cached on
first run.

## The composite metric and anti-Goodhart design

Per (original, regenerated) pair, four similarities each in [0, 1] with
higher = better:

- `s_gemini` = cosine of Gemini embeddings
- `s_dino` = cosine of DINOv2 CLS-token features
- `s_lpips` = 1 − clip(lpips_distance, 0, 1)
- `s_color` = 1 − clip(chi2_distance / 2.0, 0, 1) where chi-square is
  `0.5 * sum((p−q)² / (p+q+eps))`

Composite = mean of the four per-metric means across 20 eval images.

**Promotion gate** — a candidate is promoted only if BOTH:

1. **No-regression rule:** each individual metric stays within
   `epsilon = 0.01` of the leader's per-metric mean.
2. **Improvement rule:** `composite_candidate > composite_leader`.

If any single dimension drops by more than 0.01, candidate is rejected
even if composite improved. This defeats the most common failure mode:
boosting one signal by sacrificing another.

**3-seed re-eval:** promoted candidates re-run the full eval with 3
different generation seeds (fixed seeds 1, 2, 3 for within-session
reproducibility). The 3-seed mean must still pass the gate; otherwise
revert.

**Held-out validation:** 5 val images, `uv run harness.py --val` runs the
full pipeline without promotion logic. Run every ~10 promoted leaders.
If composite_eval climbs but composite_val flat/dropping, the strategy is
overfitting.

**VLM judge — diagnostic only, NOT optimized against:** a 6-axis
Gemini-Flash-Lite judge (subject identity, composition, lighting, palette,
style, texture; each 1–5) is logged per pair. Optimizing against it would
create a feedback loop with the prompting model. Be explicit about this.

## Eval set design

The target images are images **used in slides** (hero photos, illustrations,
icons, etc.), NOT slides themselves. 20 eval images + 5 held-out val.
Mixed sources.

Recommended distribution:
- 4× hero photography (landscape, urban, abstract, atmospheric)
- 3× conceptual flat-vector illustrations
- 3× isometric or hand-drawn illustrations
- 2× product photography on clean background
- 2× people / portraits (one solo, one group — identity stress test)
- 2× photographic metaphors (cliché stock-photo style)
- 2× background textures or patterns
- 1× UI mockup or data viz (text-rendering stress test)
- 1× icon or pictogram

Difficulty: ~6 easy, ~10 medium, ~4 hard. Val covers same archetypes
with different content (eval has flat-vector "growth chart" → val has
flat-vector "team meeting"). Once locked, neither set is modified.

## Specific design decisions to preserve in both docs

- Driver agent ≠ inner VLM. State this explicitly.
- 448×448 canonical resolution everywhere (DINOv2 patch-size compat: 448/14=32).
- `epsilon = 0.01` for the no-regression gate.
- 3-seed re-eval with fixed seeds 1, 2, 3.
- VLM-judge is logged but never optimized — explain why (feedback loop).
- Step 1 of the workflow is "re-read `logbook.md`" — surface as the single most important instruction.
- Prefer qualitatively different experiments over local optimization; call out the drift-into-one-direction failure mode.
- `IMPLEMENTATION.md` should reference `program.md` for shared definitions rather than duplicating them; cost of duplication > cost of cross-reference.
